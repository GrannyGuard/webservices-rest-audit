# Mitigatie & validatie verbeteringen

Rubric: *"Verbeteronderzoek security"* — **Mitigatie & validatie verbeteringen** (20 pt).

Dit document beschrijft per kwetsbaarheid de genomen **mitigatie**, hoe die is
**gerealiseerd** (incl. (AI-)tooling, §6) en hoe **gevalideerd** is dat het risico is
verlaagd (voor/na). De validatie steunt op drie reproduceerbare bronnen:

1. **CodeQL-herscan** (SAST) — een alert gaat van `open` → `fixed`;
2. **Unit tests** (JUnit) — bewijzen het gedrag van de fix;
3. **Build** (`mvn -B clean install`) — bevestigt dat de wijziging compileert en geen
   regressie in de bestaande tests veroorzaakt.

> Pentest-**hervalidatie** van de meest kritische bevindingen (de "na"-meting met
> dezelfde aanvalsstappen) hoort in [`05-penetration-tests`](../05-penetration-tests/)
> en wordt in de testomgeving uitgevoerd; zie ook de restpunten onderaan.

## 1. CWE-117 Log injection

**Bevinding:** [04-code-review §2.1 — F1](../04-code-review/04.md#21-bevinding-1--log-injection-cwe-117-via-rest-request-parameters)
(CodeQL `java/log-injection`, 9 alerts, `security-extended` suite — zie
[ADR-001](../02-secure-pipelines/adr-001-codeql-query-suite-scope.md)).

**Mitigatie:**
- Nieuwe gedeelde helper `RestUtil.sanitizeForLog(String)`
  (`omod-common/.../web/RestUtil.java`) verwijdert `\r`/`\n` uit user-controlled
  waarden vóórdat ze in een logregel terechtkomen — dit is het door CodeQL
  herkende sanitization-patroon voor `java/log-injection` (taint-barrier).
- Toegepast op alle 8 instromende call-sites in
  `RestUtil.getIntegerParam`, `ClearDbCacheController2_0` (×3),
  `SearchIndexController2_0` (×3) en `ConceptResource1_8`.
- **Realisatie/tooling:** geïmplementeerd met Claude Code (Sonnet 4.6) — de
  CodeQL-alertlijst (via `gh api .../code-scanning/alerts`) is gebruikt om de
  exacte bestanden/regels te identificeren; de fix zelf (één centrale sanitizer
  + toepassing op de call-sites) is handmatig gereviewd voordat gecommit.
- **Commit:** `831dafb` — `fix(security): sanitize user input before logging
  (CWE-117 log injection)`

**Validatie (voor/na):**

## 2. SQ8 / CWE-79 — Reflected XSS in Swagger debug-endpoint

**Bevinding:** [04-code-review §2.2](../04-code-review/04.md#22-bevinding-2--reflected-xss-in-swagger-debug-endpoint-cwe-79-)
(CodeQL `java/xss`, alert **#1**, severity high). `SwaggerDocController.debug()`
reflecteerde de query-parameter `tag` ongesanitized in een HTML-respons.

**Mitigatie (twee lagen):**
- **Output-encoding (de fix):** `tag` wordt nu HTML-geëncodeerd met
  `org.springframework.web.util.HtmlUtils.htmlEscape(...)` vóór het in de respons
  wordt geplaatst — dit neutraliseert de `java/xss`-taintflow.
- **Defense-in-depth (config):** de hele `/apiDocs`-boom is bovendien uitschakelbaar
  via `webservices.rest.enableSwaggerDocs` (zie
  [hardening-checklist §1](../01-security-audit/threat-model/hardening-checklist.md#1-onnodige-features-uitschakelen));
  in productie (`false`) geeft het endpoint HTTP 404 en is de vector volledig weg.
- **Realisatie/tooling:** Claude Code (Opus 4.8); de fix is een minimale,
  bibliotheek-gebaseerde encoding (geen eigen escaping) en handmatig gereviewd.

**Validatie (voor/na):**

| | Vóór | Na |
|---|---|---|
| `return "<h1>Debugging Tag: " + tag + "</h1>";` | rauwe reflectie | `... + HtmlUtils.htmlEscape(tag) + ...` |
| Payload `?tag=<script>alert(1)</script>` | uitgevoerd als script | gerenderd als tekst (`&lt;script&gt;…`) |
| CodeQL `java/xss` alert #1 | open (high) | sluit op de eerstvolgende CodeQL-herscan |

`mvn clean install` groen. Restpunt: de CI-CodeQL-run die alert #1 op `state: fixed`
zet, toevoegen als bewijsartefact (zelfde patroon als §1).

## 3. CWE-302 — Fall-through in `AuthorizationFilter` na sessie-timeout

**Bevinding:** [04-code-review §2.3](../04-code-review/04.md#23-bevinding-3--user-controlled-authentication-bypass-in-authorizationfilter-cwe-302-)
(CodeQL `java/user-controlled-bypass`, alert **#908**). Na een verlopen sessie werd
`sendError(401)` aangeroepen **zonder `return`**, waardoor de uitvoering doorliep naar
`chain.doFilter()` — de 401 werd verstuurd maar de request werd alsnog doorgegeven.

**Mitigatie:** `return;` toegevoegd direct na de `SESSION_TIMEOUT`-`sendError`
(`AuthorizationFilter.java`), zodat een timed-out request de keten daadwerkelijk stopt.

- **Realisatie/tooling:** Claude Code (Opus 4.8); één-regel-fix conform de aanbeveling
  in 04-code-review, handmatig gereviewd op neveneffecten (de "best-effort"-flow voor
  *credentials* blijft ongewijzigd — alleen de expliciete timeout-tak stopt nu).

**Validatie (voor/na):**

| | Vóór | Na |
|---|---|---|
| Verlopen sessie | 401 gezet, request loopt door naar `chain.doFilter()` | 401 gezet, keten gestopt (`return`) |
| `AuthorizationFilterLoggingTest` (12 tests) | groen | **groen** (geen regressie) |
| CodeQL `java/user-controlled-bypass` #908 | open (high) | sluit op de eerstvolgende CodeQL-herscan |

## 4. SQ9 — Host-header injectie in de Swagger-specificatie

**Bevinding:** [01-security-audit §3.5 (TM-S4 / SQ9)](../01-security-audit/01.md#35-threat-model--c4-architectuur--stride-analyse).
`SwaggerSpecificationController` nam de client-`Host`- en `X-Forwarded-Proto`-headers
ongevalideerd over in de gepubliceerde OpenAPI-spec.

**Mitigatie:**
- `RestUtil.resolveSwaggerHost(host)` valideert de `Host` tegen de allow-list
  `webservices.rest.swaggerAllowedHosts`; een niet-toegestane host valt terug op de
  canonieke (eerste) host i.p.v. de client-waarde.
- `RestUtil.sanitizeScheme(scheme, default)` staat alleen `http`/`https` toe (blokkeert
  `X-Forwarded-Proto`-injectie).
- Bedraad in `SwaggerSpecificationController`; global properties in `config.xml`. Zie
  [hardening-checklist §10](../01-security-audit/threat-model/hardening-checklist.md#10-openapi-specificatie-alleen-naar-specifieke-allowed-hosts).
- **Realisatie/tooling:** Claude Code (Opus 4.8); logica bewust in testbare helpers in
  `RestUtil` gezet i.p.v. inline in de controller.

**Validatie (voor/na):** unit tests `SwaggerHardeningRestUtilTest` — **8/8 groen**:

| Test | Bewijst |
|---|---|
| `resolveSwaggerHost_shouldReturnRequestHostWhenAllowListEmpty` | legacy-gedrag bij lege lijst |
| `resolveSwaggerHost_shouldReturnRequestHostWhenItIsOnTheAllowList` | toegestane host blijft |
| `resolveSwaggerHost_shouldFallBackToFirstAllowedHostWhenRequestHostNotAllowed` | gespoofte host → canonieke host |
| `sanitizeScheme_shouldKeepHttpAndHttps` / `…FallBackToDefaultForUnsafeScheme` | alleen http/https gereflecteerd |

`mvn clean install` groen.

## 5. Attack-surface-reductie (preventieve maatregelen)

Naast de bovenstaande code-fixes zijn **preventieve** maatregelen genomen die de
*kans* en *impact* van misbruik verlagen i.p.v. één CodeQL-alert te sluiten. Ze zijn
per stuk gedocumenteerd met status, NEN-7510-control en koppeling in de
[hardening-checklist](../01-security-audit/threat-model/hardening-checklist.md):

| Maatregel | NEN-7510 | Effect |
|---|---|---|
| Swagger/OpenAPI uitschakelbaar (`enableSwaggerDocs`) — §1 | A.8.3 | minder blootstelling SQ8/SQ9 in productie |
| Least-privilege runtime DB-user (kan nooit `DROP`) — §2 | A.8.2/A.8.3 | beperkt impact van SQL-injectie/escalatie (TM-E2) |
| Signed artifacts (cosign + SLSA-provenance + SBOM) — §8 | A.5.23/A.8.32 | verifieerbare supply chain (CD1) |
| Alle CI-actions SHA-pinned — §9 | A.5.23/A.8.32 | onveranderbare build-toolchain |

## 6. Verantwoording (AI-)tooling

De realisatie is uitgevoerd met **Claude Code (Opus 4.8)** als pair-programmer, binnen
een vaste werkwijze:

1. **Lokaliseren** — bevindingen kwamen uit CodeQL (`gh api .../code-scanning/alerts`),
   het threat model (STRIDE) en handmatige review; niet uit een blinde AI-suggestie.
2. **Genereren** — de tool stelde de fix + tests voor (centrale helpers in `RestUtil`,
   `HtmlUtils.htmlEscape`, de `return`-fix).
3. **Reviewen** — elke wijziging is handmatig beoordeeld op correctheid en
   neveneffecten vóór commit (bv. bewust géén eigen escaping maar de Spring-bibliotheek;
   bewust de credentials-flow van de filter ongemoeid laten).
4. **Valideren** — `mvn clean install` + gerichte unit tests ná elke wijziging; geen
   commit zonder groene build.

GitHub Copilot Autofix (GitHub Advanced Security) is daarnaast beschikbaar op
CodeQL-alerts (zie [04-code-review §1](../04-code-review/04.md)); suggesties zijn niet
blind overgenomen.

## 7. Kritische reflectie

- **Sterk:** de code-fixes zijn klein, bibliotheek-gebaseerd en testbaar gemaakt
  (helpers in `RestUtil` i.p.v. inline), wat reproduceerbare unit-validatie mogelijk
  maakt en de kans op regressie minimaliseert (bevestigd: bestaande filter-tests
  blijven groen).
- **Beperking — validatiediepte:** unit tests bewijzen de *logica*, niet de volledige
  HTTP-keten. De definitieve "na"-meting voor SQ8/CWE-302 is de **CodeQL-herscan** in
  CI (alert → `fixed`); die moet als artefact worden bijgevoegd, en een **pentest-hertest**
  in de testomgeving sluit de cirkel kwantitatief.
- **Beperking — gating defaultt op `true`:** `enableSwaggerDocs` staat standaard aan om
  bestaand gedrag/pentests niet te breken; de productie-winst hangt dus af van een
  correcte OTAP-configuratie (`false`). Dit is een bewuste afweging (compatibiliteit vs.
  secure-by-default) en als zodanig gedocumenteerd.
- **Beperking — least-privilege DB:** het mechanisme is aanwezig, maar de runtime-switch
  naar het DML-only account is een deploy-stap die in deze omgeving niet
  container-geverifieerd kon worden (Docker-daemon niet beschikbaar).
- **Tooling:** AI versnelde lokaliseren + boilerplate (tests), maar de
  beveiligingsbeslissingen (welke fix, welke afweging) zijn menselijk genomen en
  geverifieerd; CodeQL/unit tests — niet de AI — zijn het bewijs.
