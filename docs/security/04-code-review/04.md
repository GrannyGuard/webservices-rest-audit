# Security code review & kwetsbaarheden

Rubric: *"Verbeteronderzoek security"* — **Security code review & kwetsbaarheden** (15 pt).
NEN-7510 controls: **A.8.25** (veilige ontwikkelcyclus), **A.8.28** (veilig coderen), **A.8.29** (beveiligingstests).

---

## 1. Methodologie

De code review is gelaagd opgezet: geautomatiseerde tooling vormt de eerste verdedigingslinie en menselijke beoordeling vormt de laatste poort vóór merge.

### Geautomatiseerde tooling

**SonarCloud (SAST)**
Elke pull request triggert een SonarCloud-analyse. De pipeline is geconfigureerd als *quality gate*: een pull request waarop de quality gate faalt (bv. een nieuw security hotspot of een reliability-issue met rating slechter dan A) kan niet gemerged worden totdat het probleem handmatig is opgelost of expliciet gedocumenteerd afgewogen. Dit dwingt af dat bevindingen niet stilzwijgend passeren.

**CodeQL (SAST, query-suite `security-extended`)**
GitHub Advanced Security voert bij elke push een CodeQL-scan uit met de `security-extended` query-suite (zie [ADR-001](../02-secure-pipelines/adr-001-codeql-query-suite-scope.md) voor de keuzeonderbouwing). Alerts verschijnen als *code scanning alerts* in de GitHub Security-tab. De organisatie-instelling blokkeert merge bij nieuwe hoog-risico-alerts totdat deze worden opgelost of als `false positive` gedismisst met onderbouwing.

**SBOM & CVE-check (SCA)**
Bij elke pull request wordt een CycloneDX SBOM gegenereerd en gecontroleerd op bekende CVE's (via Dependency-Review Action / Snyk). Een pull request met afhankelijkheden die een CVE bevatten boven de drempelwaarde faalt automatisch. Dit vereist handmatige actie — updaten van de afhankelijkheid of een gedocumenteerde risicoafweging — voordat merge mogelijk is.

**GitHub Advanced Security — AI-ondersteunde review (Copilot Autofix)**
GitHub's ingebouwde AI-tooling (Copilot Autofix) analyseert CodeQL-alerts en genereert suggesties voor mitigatie rechtstreeks als PR-commentaar. Deze suggesties zijn niet blindelings overgenomen: elke autofix-suggestie is handmatig beoordeeld op correctheid en contextrelevantie voordat deze werd toegepast of verworpen.

### Verplichte menselijke review

Op organisatieniveau is *required reviewers* afgedwongen: elke pull request vereist minimaal één goedkeuring van een ander teamlid voordat merge is toegestaan. Dit voorkomt dat bevindingen van de geautomatiseerde tooling worden omzeild zonder bewust menselijk oordeel. De combinatie van automatische gate (SonarCloud, CodeQL, SBOM) én menselijke goedkeuring zorgt dat kwetsbaarheden die door tooling worden gevlagd altijd een expliciete beslissing vereisen.

### Handmatig onderzochte bestanden

Naast de geautomatiseerde scans zijn de volgende klassen handmatig gereviewed, op basis van het aanvalsoppervlak-overzicht (Sprint 3) en de risicomatrix (Sprint 2):

- `AuthorizationFilter.java` — toegangscontrole en authenticatieverwerking
- `SessionController1_9.java` — sessiebeheer en inlogstroom
- `BaseDelegatingResource.java` — centrale resource-routing met brede rechtenimpact

## 2. Bevindingen

De CodeQL-scan met de `security-extended`-suite (scan `e476489`, 2026-06-10) levert —
na het wegfilteren van de SCA/Snyk-*dependency*-alerts, die in
[SBOM & CVE-advies (§03)](../03-sbom-en-cve-advies/03.md) thuishoren — **7 open**
code-niveau security-bevindingen op (F2–F6), naast de **9 reeds gemitigeerde**
`java/log-injection`-alerts (F1) en **4 gedismíste** alerts (F1-restpunt). SonarCloud
levert aanvullende kwaliteits-/reliability-hotspots (zie §4 en de SQ1–SQ6-reeks in
[`01-security-audit/01.md §4`](../01-security-audit/01.md#4-risico-evaluatie)).
Aanvullend zijn via **handmatige review** drie bevindingen geïdentificeerd die niet
door statische analyse worden opgepikt (F7–F9): session fixation, ontbrekende
brute-force bescherming, en het onbeveiligde `/session/diag`-endpoint.

Elke bevinding hieronder is **handmatig geverifieerd tegen de broncode** (bestand +
regel + CodeQL-meldtekst). Per bevinding is naast de CodeQL-`security-severity` een
expliciete **eigen reële-impact-inschatting** opgenomen: een hoge CodeQL-ernst betekent
niet automatisch hoge praktijkimpact, en die nuance — vereist voor het rubric-onderdeel
*"goed onderbouwde risico-inschattingen en duidelijke relatie kwetsbaarheid↔systeemgebruik"*
— is per geval onderbouwd.


### 2.1 Bevinding 1 — Log injection (CWE-117) via REST-request-parameters

- **Bron:** CodeQL `java/log-injection`, query-suite `security-extended` (zie
  [ADR-001](../02-secure-pipelines/adr-001-codeql-query-suite-scope.md))
- **Bestanden + regels (vóór fix, commit `8521cdc`):**
  - `omod-common/.../web/RestUtil.java:521`
  - `omod/.../v1_0/controller/openmrs2_0/ClearDbCacheController2_0.java:79,86,95`
  - `omod/.../v1_0/controller/openmrs2_0/SearchIndexController2_0.java:77,81 (×2)`
  - `omod/.../v1_0/resource/openmrs1_8/ConceptResource1_8.java:567`
- **Beschrijving:** de REST-controllers loggen request-afgeleide waarden
  (`resource`, `subResource`, `uuid`, query-parameters) direct in `log.debug()`/
  `log.info()`/`log.error()`-statements zonder neutralisatie. Een aanvaller kan
  CR/LF-tekens (`\r\n`) in deze velden opnemen om extra, valse logregels te
  injecteren ("log forging") — bv. een nep `INFO`-regel die een succesvolle login
  suggereert, om forensisch onderzoek te misleiden.
- **OWASP Top 10 (2021):** A09:2021 – Security Logging and Monitoring Failures
  (gecombineerd met A03:2021 – Injection als onderliggend mechanisme).
  Referentie: [OWASP — Log Injection](https://owasp.org/www-community/attacks/Log_Injection),
  [CWE-117](https://cwe.mitre.org/data/definitions/117.html).
- **NEN-7510 control:** A.8.15 (Logging) — de integriteit van het audit-logbestand
  (KJ6, zie [01-security-audit §2.1](../01-security-audit/01.md)) is een
  voorwaarde voor bruikbaarheid van logging als bewijs/forensisch middel.
- **Risico bij niet-oplossen:** een vervalst auditlog ondermijnt elk forensisch
  onderzoek na een incident — precies het scenario dat NEN-7510 A.8.15 en de
  audit-trail-eis (zie kroonjuweel KJ6) moeten voorkomen. CWE-117 wordt door
  OWASP geclassificeerd binnen A09, een categorie die volgens de OWASP Top 10
  2021 in >90% van de onderzochte applicaties aanwezig was.
- **Risicoscore:** **CQL1** in risicomatrix — score **12** 🟠 Midden, KJ6. Zie [01.md §4.2](../01-security-audit/01.md#42-risicomatrix).
- **Status:** ✅ **Gemitigeerd en geverifieerd** in commit `831dafb`
  ("fix(security): sanitize user input before logging (CWE-117 log injection)").
  Alle 9 oorspronkelijke `java/log-injection`-alerts (incl. `AuthorizationFilter.java:108`)
  staan na de scan op `main` (`e476489`, 2026-06-10) op `state: fixed`. Zie
  [Mitigatie & validatie §1](../06-mitigatie-en-validatie/06.md#1-cwe-117-log-injection)
  voor de volledige voor/na-tabel.
- **Restpunt — 4 als "false positive" gedismíste alerts (kritische auditnoot):**
  dezelfde logging-herschrijving voor issue #65 (commit `8521cdc`) introduceerde
  4 *nieuwe* `java/log-injection`-alerts in `AuthorizationFilter.java`
  (regels 72, 84, 115, 122 — alerts `808`–`811`), op 2026-06-08 door @BaasW
  gedismissed als **"false positive"** zonder toelichting. **Audit-observatie:** deze
  classificatie is **niet volledig onderbouwd**. De betreffende statements loggen
  wél user-controlled waarden — de request-URI (`getRequestURI()`) en de aangeleverde
  Basic-Auth-gebruikersnaam (`attemptedUser`, regels 115/122). Dat ze SLF4J-*parameterized*
  logging (`{}`) gebruiken neutraliseert CRLF-injectie **niet** automatisch: of een
  `\r\n` in de gebruikersnaam een valse logregel oplevert, hangt af van de
  log-appender/encoder-configuratie (bv. log4j2 `%encode{}` / `%replace`). Voor een
  correcte "false positive"-classificatie moet daarom worden vastgelegd dat de
  productie-logconfiguratie newlines escapet — is dat niet aantoonbaar, dan is
  sanitisatie (zoals bij de 9 gefixte alerts) alsnog vereist. Zie
  [Mitigatie & validatie §1](../06-mitigatie-en-validatie/06.md#1-cwe-117-log-injection).

### 2.2 Bevinding 2 — Reflected XSS in Swagger debug-endpoint (CWE-79) 🔴

- **Bron:** CodeQL `java/xss` (alert **#1**, severity **high**, `security-extended`);
  onafhankelijk geïdentificeerd als **SQ8** in de threat-modelling (§3.5) en als
  *high risk* in [attack-surface §4](../01-security-audit/threat-model/attack-surface.md#4-categorie-c--module-admin-controllers).
- **Bestand + regel:** `omod/.../web/controller/SwaggerDocController.java:27`
- **Beschrijving:** de `debug`-handler retourneert de request-parameter `tag`
  ongesanitized in een HTML-respons:
  ```java
  return "<h1>Debugging Tag: " + tag + "</h1>";
  ```
  CodeQL-meldtekst: *"Cross-site scripting vulnerability due to a user-provided value."*
  Een aanvaller kan `?tag=<script>…</script>` injecteren. Het endpoint
  `/module/webservices/rest/apiDocs/debug` valt **buiten** `AuthorizationFilter`
  (categorie C, TB6) — de XSS is dus **zonder authenticatie en zonder IP-allowlist**
  bereikbaar.
- **OWASP Top 10 (2021):** A03:2021 – Injection (XSS valt sinds 2021 onder Injection).
  Referentie: [OWASP — XSS](https://owasp.org/www-community/attacks/xss/),
  [CWE-79](https://cwe.mitre.org/data/definitions/79.html).
- **NEN-7510 control:** A.8.28 (veilig coderen — output-encoding), A.8.26
  (application security requirements).
- **Risico bij niet-oplossen:** reflected XSS uitgevoerd in de browser van een ingelogde
  admin/zorgverlener kan sessiecookies of CSRF-tokens stelen → overname van een
  geprivilegieerde sessie (KJ3) en van daaruit toegang tot KJ1/KJ2. Omdat het endpoint
  publiek en onauthenticated is, volstaat één crafted link (phishing) — geen voorafgaande
  toegang vereist.
- **Risicoscore:** **SQ8** — score **12** 🟠 Midden ⚠️ (raakt KJ3), zie
  [01.md §4.2](../01-security-audit/01.md#42-risicomatrix).
- **Status:** 🔴 **Open (code-fix)** — output-encoding nog niet toegepast (alert #1).
  **Wel beperkt op configuratieniveau:** de Swagger-endpoints (incl. `/apiDocs/debug`)
  zijn nu uitschakelbaar via de global property `webservices.rest.enableSwaggerDocs`
  (default `true`; in productie op `false` → HTTP 404), wat de blootstelling van deze
  XSS in productie wegneemt — geïmplementeerd, zie
  [hardening-checklist §1](../01-security-audit/threat-model/hardening-checklist.md#1-onnodige-features-uitschakelen).
  Aanbevolen definitieve fix: HTML-encode `tag` (bv. Spring `HtmlUtils.htmlEscape`)
  **of** het debug-endpoint verwijderen — het heeft geen productiefunctie.

### 2.3 Bevinding 3 — User-controlled authentication bypass in `AuthorizationFilter` (CWE-302) 🔴

- **Bron:** CodeQL `java/user-controlled-bypass` (alert **#908**, severity **high**,
  `security-extended`) + handmatige review.
- **Bestand + regels:** `omod-common/.../web/filter/AuthorizationFilter.java:113`
  (`Context.authenticate(...)`) en `:86` (ontbrekende `return`).
- **Beschrijving:** CodeQL-meldtekst: *"Sensitive method may not be executed depending on
  a this condition, which flows from user-controlled value."* Of `Context.authenticate(...)`
  wordt aangeroepen, hangt volledig af van door de gebruiker gestuurde condities
  (aanwezigheid/vorm van de `Authorization`-header). De filter is bovendien **by design
  best-effort** — het klassecommentaar (regels 34-35) stelt expliciet dat de filter
  *"never stops execution"* en op de API vertrouwt om alsnog te falen. **Aanvullend
  handmatig gevonden:** na een verlopen sessie wordt op regel 86 `sendError(401)`
  aangeroepen **zonder `return`**, waardoor de uitvoering doorloopt naar
  `chain.doFilter()` (regel 131) — de 401 wordt verstuurd maar de request wordt alsnog
  doorgegeven aan de keten.
- **OWASP Top 10 (2021):** A07:2021 – Identification and Authentication Failures
  (samenhangend met A01:2021 – Broken Access Control). Referenties:
  [CWE-302](https://cwe.mitre.org/data/definitions/302.html) (Authentication Bypass by
  Assumed-Immutable Data), [CWE-287](https://cwe.mitre.org/data/definitions/287.html).
- **NEN-7510 control:** A.8.5 (veilige authenticatie), A.8.3 (toegangsbeveiliging).
- **Risico bij niet-oplossen:** de combinatie "filter blokkeert nooit" + ontbrekende
  `return` betekent dat toegangscontrole volledig leunt op de aanname dat élke downstream
  resource zélf een privilege-check doet (TM-E1/TM-E3). Waar die check ontbreekt — de
  categorie-B-endpoints zónder eigen check uit
  [attack-surface §3](../01-security-audit/threat-model/attack-surface.md#3-categorie-b--version-specific-controllers) —
  is een ongeauthenticeerde of verlopen request niet hard geblokkeerd.
- **Risicoscore:** gekoppeld aan **SQ1** (kwetsbare sessie-validatie, score **15**
  🔴 Hoog) en het systemische patroon A.8.3 (declaratieve access control, prio 11 in
  [01.md §5.1](../01-security-audit/01.md)).
- **Status:** 🔴 **Open** (alert #908). Aanbevolen fix: `return` toevoegen na de
  `SESSION_TIMEOUT`-`sendError` (regel 86); op termijn declaratieve autorisatie i.p.v.
  de best-effort filter.

### 2.4 Bevinding 4 — Mogelijk gevoelige data in log (CWE-532) 🟡

- **Bron:** CodeQL `java/sensitive-log` (alert **#4**, severity **high**, `security-extended`).
- **Bestand + regel:** `omod/.../v1_0/helper/LayoutTemplateProvider.java:124`
- **Beschrijving:** `log.warn("Found invalid token while parsing GlobalProperty " +
  propertyName + " : " + token);` — de waarde `token` komt uit de inhoud van een global
  property (`customDefaults`, gesplitst op `,` en `=`) en wordt via string-concatenatie
  gelogd. CodeQL-meldtekst: *"This potentially sensitive information is written to a log file."*
- **OWASP Top 10 (2021):** A09:2021 – Security Logging and Monitoring Failures.
  Referentie: [CWE-532](https://cwe.mitre.org/data/definitions/532.html) (Insertion of
  Sensitive Information into Log File).
- **NEN-7510 control:** A.8.15 (logging), A.5.34 (privacy en bescherming PII).
- **Reële-impact-inschatting:** in déze context bevat de gelogde global property
  *adres-layout-defaults* — geen credentials of PII — dus de **reële gevoeligheid is laag**.
  Het patroon (rauwe global-property-inhoud loggen) blijft wel onwenselijk: niet elke
  global property is per definitie niet-gevoelig (vgl. SQ7, waar global properties
  database-/mailcredentials kunnen bevatten).
- **Status:** 🟡 **Open — lage prioriteit** (alert #4). Aanbevolen: log alleen de
  *property-naam* en het feit dát een token ongeldig is, niet de tokenwaarde zelf.
  Kandidaat voor "accepteren met onderbouwing", mits gedocumenteerd.

### 2.5 Bevinding 5 — Integer overflow/underflow op paginatieparameters (CWE-190/191) 🟡

- **Bron:** CodeQL `java/tainted-arithmetic` (alerts **#5** en **#6**, severity **high**,
  `security-extended`).
- **Bestand + regels:** `omod-common/.../web/RequestContext.java:172` en `:183`
- **Beschrijving:** de paginatieparameters `startIndex` en `limit` (query-parameters,
  user-controlled) gaan ongevalideerd de rekenkunde in:
  - regel 172: `query += … + (startIndex + limit);` — **overflow** (CodeQL: *"…potentially
    causing an overflow"*); geen bovengrens-controle.
  - regel 183: `int prevStart = startIndex - limit;` — **underflow**, maar dit wordt
    direct daarna geklemd (`if (prevStart < 0) prevStart = 0;`, regels 184-185).
- **OWASP Top 10 (2021):** A04:2021 – Insecure Design. Referenties:
  [CWE-190](https://cwe.mitre.org/data/definitions/190.html) (overflow),
  [CWE-191](https://cwe.mitre.org/data/definitions/191.html) (underflow).
- **NEN-7510 control:** A.8.28 (veilig coderen — invoervalidatie).
- **Reële-impact-inschatting:** **laag**. Het resultaat wordt enkel gebruikt om de
  `next`/`prev`-hyperlink-string te bouwen; een overflow levert hooguit een onjuiste
  `startIndex` in een paginatielink op — geen geheugen-/databasecorruptie (Java-integers
  wrappen veilig, geen out-of-bounds). Regel 183 is bovendien al geklemd. Wel correct om
  `startIndex`/`limit` server-side te begrenzen.
- **Status:** 🟡 **Open — lage prioriteit** (alerts #5, #6). Aanbevolen: begrens
  `startIndex`/`limit` op een maximum vóór de rekenkunde (sluit aan op de bestaande
  `MAX_RESULTS_ABSOLUTE`-instelling).

### 2.6 Bevinding 6 — Te ruime tekenreeks-range in log-parser-regex (CWE-20) 🟡

- **Bron:** CodeQL `java/overly-large-range` (alerts **#2** en **#3**, severity **medium**,
  `security-extended`).
- **Bestand + regel:** `omod-common/.../helper/ServerLogActionWrapper.java:70`
- **Beschrijving:** de regex voor het parsen van serverlog-regels bevat `[A-z]`:
  ```
  (INFO|ERROR|WARN|DEBUG)\s.*?[-].*?\s((?:[A-z][A-z].+))\s[|](.*?)[|]\s((.*\n*)+)
  ```
  CodeQL-meldtekst: *"Suspicious character range that is equivalent to
  [A-Z[\]^_`a-z]."* De range `A-z` omvat naast letters ook zes leestekens
  (`[ \ ] ^ _ \``, ASCII 91–96). Dit is primair een **correctheids-/kwaliteitsbug** in
  de log-parser (`/serverlog`-endpoint). Secundair: het deelpatroon `((.*\n*)+)` heeft
  een geneste quantifier — een vorm die gevoelig kan zijn voor catastrofaal backtracking
  (ReDoS) bij speciaal vervaardigde invoer.
- **OWASP Top 10 (2021):** A04:2021 – Insecure Design. Referenties:
  [CWE-20](https://cwe.mitre.org/data/definitions/20.html) (improper input validation);
  de ReDoS-vorm raakt [CWE-1333](https://cwe.mitre.org/data/definitions/1333.html).
- **NEN-7510 control:** A.8.28 (veilig coderen).
- **Reële-impact-inschatting:** **lage security-impact** — de parser verwerkt
  server-gegenereerde logregels (geen directe user-input), dus de ReDoS-vector is beperkt
  tot een actor die de serverlogs al kan beïnvloeden. De `[A-z]`-bug kan hooguit
  logger-namen onjuist groeperen. Kandidaat voor laagprioriteit-fix (`[A-Za-z]` i.p.v.
  `[A-z]`; geneste quantifier herzien) of "accepteren met onderbouwing".
- **Status:** 🟡 **Open — lage prioriteit** (alerts #2, #3).

### 2.7 Bevinding 7 — Session Fixation (CWE-384) via ontbrekende sessie-regeneratie na login

- **Bron:** Handmatige review van `AuthorizationFilter.java` — niet detecteerbaar door
  statische analyse (SAST vindt geen aanwezig patroon, maar de *afwezigheid* van
  `session.invalidate()` na login). Gerelateerd aan SQ1 in risicomatrix.
- **Bestanden + regels:**
  - `omod-common/.../web/filter/AuthorizationFilter.java:82` —
    `getRequestedSessionId()` geeft het door de *client* aangeleverde sessie-ID terug
    zonder server-side validatie.
  - `AuthorizationFilter.java:113` — na `Context.authenticate()` wordt het bestaande
    sessie-object *niet* geïnvalideerd; een sessie-ID die een aanvaller vóór authenticatie
    heeft "geplant" blijft na login geldig.
- **Beschrijving:** session fixation stelt een aanvaller in staat een bekende sessie-ID
  vóór het inloggen van het slachtoffer te planten. Zodra het slachtoffer inlogt en het
  sessie-ID niet wordt ververst, kan de aanvaller met diezelfde ID de geauthenticeerde
  sessie overnemen zonder ooit een wachtwoord te kennen.
- **OWASP Top 10 (2021):** A07:2021 – Identification and Authentication Failures.
  Referentie: [OWASP — Session Fixation](https://owasp.org/www-community/attacks/Session_fixation),
  [CWE-384](https://cwe.mitre.org/data/definitions/384.html).
- **NEN-7510 control:** A.8.5 (Secure Authentication).
- **Risico bij niet-oplossen:** sessie-overname geeft volledige toegang tot patiëntdossiers
  (KJ1, KJ2) met de rechten van het slachtoffer — direct datalek met AVG Art. 33 meldplicht
  en risico op verkeerde zorgbeslissingen door gemanipuleerde dossiers.
- **Relatie met systeemgebruik:** OpenMRS verwerkt bijzondere persoonsgegevens (AVG Art. 9,
  WGBO Art. 7:454). Elke geauthenticeerde sessie geeft toegang tot diagnoses, medicatie en
  patiëntidentificatie. In combinatie met F2 (XSS) vormt dit een complete aanvalsketen:
  XSS steelt sessie-cookie → aanvaller neemt sessie over.
- **Status:** ❌ **Open** — geen mitigatie doorgevoerd; zie prioriteit 5 in
  [01-security-audit §5.1](../01-security-audit/01.md#51-prioritering-non-compliancies).

### 2.8 Bevinding 8 — Ontbrekende brute-force bescherming op authenticatie-endpoint (CWE-307)

- **Bron:** Handmatige review van `AuthorizationFilter.java` — afwezigheid van een
  mechanisme, niet detecteerbaar door SAST. Gap A.8.5 in
  [01-security-audit §3.2](../01-security-audit/01.md#32-a85--authenticatie-secure-authentication).
- **Bestand + regel:** `omod-common/.../web/filter/AuthorizationFilter.java:113` —
  `Context.authenticate(userAndPass[0], userAndPass[1])` aangeroepen zonder rate-limiting,
  account-lockout of throttling. Een aanvaller kan onbeperkt authenticatiepogingen doen.
- **Beschrijving:** de filter stuurt elke inkomende Basic Auth-aanvraag direct door naar
  `Context.authenticate()` zonder enig mechanisme dat het aantal pogingen beperkt.
  Geautomatiseerde aanvallen (credential stuffing, dictionary attack, brute force) zijn
  triviaal uitvoerbaar.
- **OWASP Top 10 (2021):** A07:2021 – Identification and Authentication Failures.
  Referentie: [OWASP Testing Guide OTG-AUTHN-003](https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/04-Authentication_Testing/03-Testing_for_Weak_Lock_Out_Mechanism),
  [CWE-307](https://cwe.mitre.org/data/definitions/307.html).
- **NEN-7510 control:** A.8.5 (Secure Authentication).
- **Risico bij niet-oplossen:** een gecompromitteerd klinisch account geeft toegang tot alle
  patiëntdossiers van die gebruiker (KJ1, KJ2, AVG Art. 9). OWASP ASVS Level 1 vereist
  account-lockout als minimumvereiste.
- **Relatie met systeemgebruik:** in een ziekenhuisomgeving met veelgebruikte zwakke
  wachtwoorden (naam + geboortejaar, afdelingsnaam) is een dictionary attack realistisch en
  snel succesvol, met direct risico voor patiëntveiligheid.
- **Status:** ❌ **Open** — geen mitigatie doorgevoerd; zie prioriteit 2 in
  [01-security-audit §5.1](../01-security-audit/01.md#51-prioritering-non-compliancies).

### 2.9 Bevinding 9 — Unauthenticated `/session/diag` endpoint lekt rollen en privileges (CWE-200)

- **Bron:** Handmatige review van `SessionController1_9.java`; gap A.8.3 in
  [01-security-audit §3.1](../01-security-audit/01.md#31-a83--toegangsbeveiliging-access-restriction).
- **Bestand + regels:** `omod/.../v1_0/controller/openmrs1_9/SessionController1_9.java:172–187` —
  `@RequestMapping(value = "/diag")` zonder autorisatiecheck. De Javadoc luidt expliciet:
  *"NOTE: No authorization check — accessible to any caller (authenticated or not)."*
  Bij een actieve sessie retourneert het endpoint `userRoles` (r. 184) en `userPrivileges`
  (r. 185).
- **Beschrijving:** elke caller — ook anoniem — kan het endpoint aanroepen. Bij een actieve
  sessie lekt het de volledige rol- en privilege-set. In combinatie met F7 (session fixation)
  geeft dit een aanvaller bevestiging dat de gecompromitteerde sessie de gewenste rechten bevat.
  Logging is toegevoegd (Sprint 3: `log.warn("DIAG_ACCESS ...")`), maar het endpoint is niet
  beveiligd.
- **OWASP Top 10 (2021):** A01:2021 – Broken Access Control.
  Referentie: [CWE-200](https://cwe.mitre.org/data/definitions/200.html).
- **NEN-7510 control:** A.8.3 (Toegangsbeveiliging).
- **Risico bij niet-oplossen:** een aanvaller kan vooraf de rechten-structuur van accounts
  verkennen (KJ4 — autorisatiegegevens), wat gerichte aanvallen op hoog-geprivilegieerde
  accounts verlaagt. CWE-200 valt onder A01:2021 — de meest voorkomende
  kwetsbaarheidscategorie in de OWASP Top 10 2021 (94% van geteste applicaties).
- **Relatie met systeemgebruik:** rollen en privileges zijn in een ziekenhuis directe
  informatie over de organisatiestructuur en bevoegdheden van medewerkers. Blootstelling
  aan anonieme callers schendt het principle of least privilege en biedt aanvallers een
  blauwdruk voor privilege-escalation op het medisch dossier-systeem.
- **Status:** ⚠️ **Gedeeltelijk gemitigeerd** — logging toegevoegd in Sprint 3 (commit
  `8521cdc`), endpoint zelf heeft nog geen autorisatiecheck. Zie prioriteit 3 in
  [01-security-audit §5.1](../01-security-audit/01.md#51-prioritering-non-compliancies).
